# Phase 16/16 — QA (Aceite do MVP)

> Última phase do fluxo de execução do **cs2-lab**. Ao concluir, o MVP está pronto.
>
> **Antes de começar, leia:** o **Protocolo de execução** e as **Convenções técnicas** em [`00-indice.md`](00-indice.md) e o contrato de dados em [`_shared/data-contract.md`](_shared/data-contract.md). Leia também `AGENTS.md` e `docs/architecture.md`.

## Contexto

O **cs2-lab** é uma plataforma local e pessoal de análise de demos de Counter-Strike 2 (backend Python/FastAPI + frontend Next.js), usada offline para revisar partidas já gravadas. **Regra inviolável:** nada de leitura de memória do jogo, DMA, injeção de código, wallhack, radar ao vivo, automação de gameplay ou qualquer vantagem competitiva em tempo real — apenas analytics e revisão de demos `.dem` gravadas. Você atua como **QA/DevOps engineer**.

**Stack relevante a esta phase:** pytest (backend), npm lint/build (frontend), docker-compose (**PostgreSQL + Redis**).

**Onde os arquivos vivem:** `backend/app/tests/`, `README.md`, `AGENTS.md`, `docker-compose.yml`, raiz do projeto.

## Pré-requisitos

- Todas as phases anteriores (01–15) concluídas.

## Objetivo

Validar o MVP de ponta a ponta, garantir testes/lint/build, finalizar documentação e corrigir erros críticos.

## Escopo (o que fazer)

- Garantir que o `docker-compose.yml` sobe **PostgreSQL + Redis** e que o backend conecta.
- Rodar e corrigir:
  ```bash
  docker compose up -d postgres redis
  cd backend && alembic upgrade head && pytest
  cd frontend && npm run lint && npm run build
  ```
- Garantir testes unitários mínimos no backend e lint/format configurados.
- Garantir `README.md` (instalação e uso) e `AGENTS.md` completos.
- Validar fluxo mínimo com dados mockados: importar demo/fixture → status de parsing → abrir partida → relatório → replay 2D → ao menos um heatmap.
- Corrigir erros críticos encontrados.

### Critérios de aceite do MVP (checklist global)

- [ ] Backend sobe localmente (com PostgreSQL + Redis via docker-compose).
- [ ] Frontend sobe localmente e abre no navegador.
- [ ] É possível importar uma demo ou usar fixture mock; a demo aparece na lista com status correto.
- [ ] É possível abrir uma partida e ver o relatório básico.
- [ ] É possível abrir o replay 2D e ver mapa + jogadores se movendo.
- [ ] É possível ver pelo menos um heatmap básico, incluindo o mapa de movimento de um player.
- [ ] Stats de utility (granadas/flashes), kill matrix e posição de morte (`top_spot`) do player aparecem.
- [ ] Sucesso por bombsite (win rate de round por site) aparece no relatório.
- [ ] Stats por arma por player (disparos, accuracy, kills) aparecem.
- [ ] Código organizado; há `README.md`, `AGENTS.md` e testes mínimos.
- [ ] Nada depende de serviço externo obrigatório além do PostgreSQL/Redis locais (containerizados).
- [ ] Nenhum recurso de cheat, leitura de memória, injeção, DMA ou vantagem competitiva ao vivo foi implementado.
